# stay-curation-analytics

감성 숙소 큐레이팅 서비스의 실데이터 기반 분석 프로젝트.
비즈니스 이해 → KPI 설계 → 퍼널/코호트 분석 → A/B 테스트 설계·검증까지, 분석가로서 스스로 기획하고 검증하는 전 과정을 목표로 한다.

> 2인 협업 프로젝트이며, GitHub 브랜치/PR 기반 협업 워크플로 습득도 목표 중 하나입니다.
> 프로젝트 방향에 대한 고민과 의사결정 기록은 [docs/ideas.md](docs/ideas.md)에 있습니다.

## 프로젝트 목표

1. **비즈니스 이해** — 서비스 운영자 인터뷰를 통해 수익 구조와 핵심 지표 파악
2. **KPI 트리 설계** — 매출/성장을 구성 지표로 분해하고 병목 지점을 데이터로 확인
3. **퍼널 · 코호트 분석** — 유입 → 상세 조회 → 예약 전환 흐름 분석
4. **A/B 테스트 설계 및 검증** — 가설 수립, 표본 크기 계산, 검정력 분석, 결과 해석
5. **실제 개선 제안** — 분석 결과를 운영자에게 제안하고 반영 여부까지 추적

## 레포 구조

```
├── data/          # 원본/가공 데이터 (git에 올리지 않음, .gitignore 처리)
│   ├── raw/
│   └── processed/
├── notebooks/     # 탐색적 분석 노트북 (파일 단위로 담당 분리)
├── src/           # 재사용 가능한 파이썬 모듈 (전처리, 지표 계산 등)
├── reports/       # 분석 결과 정리 (마크다운/이미지)
└── docs/          # 기획 문서, 아이디어, 협업 규칙
```

## 🔒 데이터 보안 (위반 절대 불가)

이 프로젝트는 **실제 서비스의 실데이터**를 다루며, 레포는 **public**입니다.

- **원본 데이터 파일은 어떤 경우에도 커밋 금지.** 각자 로컬 `data/`에만 보관.
- 한 번 push된 데이터는 삭제 커밋으로도 지워지지 않고 이력에 영구히 남습니다.
- 노트북 출력 셀의 개인정보도 유출입니다 — 클론 직후 `nbstripout` 설치 **필수**.
- 레포에 공개 가능한 것은 코드 · 스키마 · **집계된** 결과 수치뿐입니다.
- 상세 규칙: [docs/collaboration.md](docs/collaboration.md)의 "실데이터 보안" 섹션 **필독**.

## 협업 규칙 (필독)

**[docs/collaboration.md](docs/collaboration.md)를 클론 직후, 첫 커밋 전에 반드시 끝까지 읽을 것.**

🚨 절대 금지 5계명 요약:

1. `main` 직접 커밋/푸시 금지 — 항상 브랜치 → PR → 상대 승인 → 머지
2. `git push -f` (force push) 금지 — 상대방의 이력이 날아감
3. `pull` 없이 작업 시작 금지 — 충돌의 90%가 여기서 발생
4. 같은 노트북(.ipynb) 동시 수정 금지 — 한 파일 = 한 사람
5. 데이터 파일 커밋 금지 — 위 보안 섹션 참고
